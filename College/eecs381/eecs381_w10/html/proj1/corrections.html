Ordered_container.h, last paragraph in initial block comment about validity of pointer values. Client code may want the data_ptr to be NULL.

Using assert() throughout the Ordered_container code is a good way to help you catch your programming errors if you accidentally give the OC functions a NULL pointer to a container or item. However, the client code might want to hand in a NULL pointer to a data object - it can do so as long as it also supplies an ordering function that knows what to do with a NULL data object pointer - e.g. always put it at the beginning of the container or something like that. So your Ordered_container code should not make an assertion that a data pointer supplied to Ordered_container must be non-NULL - that's the client's business.


Important: In general, the autograder tests are not supposed to test for undefined behavior, because it is, well, undefined - what would the correct behavior be? This means that the autograder will not test for what happens if we give your OC functions a NULL pointer to a container. This correction was necessary to clarify that giving the OC functions a NULL data pointer is a valid possibility as far as Ordered_container is concerned.


Ordered_container.h, comment for OC_find_item_arg. Supplied comparison function must define a ordering consistent with the container ordering.

The purpose of this function is to allow you to do the same kind of search as OC_find_item(), but more conveniently. That is, you can supply just the key value of the sought-for item instead of creating a dummy "probe" data object containing the key value you are interested in. Thus the ordering implied by the supplied comparison function is assumed to be consistent with the ordering used for OC_insert() and OC_find_item(). This allows for example, binary search to be used for both find functions in the array implementation. The effects of violating this assumption are not defined.


Ordered_container.h, at end of 3rd paragraph in initial block comment. Duplicate items not allowed.

Add the following:

Each data object pointed to by an item in the container is assumed to be unique in the container. That is, there are not supposed to be two items in the container whose data objects are equal according to the comparison function. The results of violating this assumption are undefined.

Explanation:

Because the interface is so restricted, having duplicate data objects in the container would be extremely clumsy and difficult to work with, which is beside the point of the project. This restriction is not implemented in the container code itself, but rather by never adding an item to the container without first checking to be sure a matching item is not already in the container.


P. 7, under Ordered_array. Item pointer for array implementation cannot contain an index value.

Change:

The cells of the array are designated to the client code by casting the address or index of a cell to/from a void *.

To:

The cells of the array are designated to the client code by casting the address of a cell to/from a void *.

Explanation:

The index (subscript) of a cell can't be interpreted without the starting address of the array. The function OC_get_data_ptr takes only a single parameter, the item pointer, so there is no way for it to get the additional information needed to use a subscript value packed into the item pointer.
