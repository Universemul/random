P. 1, Option S2. Statistics View contents different from status/describe() output.

The output produced by the "status" command is what is meant by "statistics" in this context. But the Statistic View is not simply the output of the describe() function.  Rather, your Statistics View is supposed to fit in with the two existing view types in the Model-View-Controller pattern specified in Project 5, and so must receive updates along the same lines as the P4/P5 views. Therefore, add this restriction to the description of Option S2:


Add after the first sentence in Option S2:

Your design and implementation of Statistics View is not allowed to call or depend on the describe() function. Your Statistics View must show the statistics information in a different layout than describe() produces, and you are free to choose to show only an interesting subset of the statistics. 


P. 1, Option S2. Statistics View: A bit more about the design goal.

After:

This must be done in a way that not only is compatible with the existing views, but will also allow creating new views that have mixtures of the previous location information and the new statistics information. 

Add:

Examples: A bridge-like view in which each ship is displayed in a way that shows how much protection it has - something that might be handy in combat. A map-like view in which Tankers and Cruise Ships are displayed in a way that you can tell what their states are: inbound to loading, not on a cruise, etc.  You are not required to implement such combination views, but your design should make them relatively easy to implement. That is, if your Statistics View shows the protection and these states, it should be possible to create these example combination views without touching the code for Model or the Sim_object family, and only adding the code for the combination view class itself.
