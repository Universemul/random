Record skeleton header, specification comment for constructor with a stream argument. Specification for reading input into variables is too vague.

The specification in question is:

// Where appropriate, input is read directly into member variables.


Now,if you use getline to read the title from the file directly into the String member variable for the title, the member variable will have a final allocation resulting from the doubling rule. This means that the total String allocation will be different (larger) than if you use getline to read the title from the file into a local String variable and then assign it to the title member variable.


The instructor's solution used a general-purpose title reading function that skips leading whitespace, checks the contents read by getline to eliminate extra spaces, and finally returns the cleaned up title by value, and then stores that in the member variable, resulting in the title member variable occupying minimum allocation. Perhaps this is overkill, but reading the title is such a cranky process that I thought I should put in a function, overlooking that this was inconsistent with the specification which was also vague - "Where appropriate" - what does this mean?


Rather than change the posted samples and autograder tests at this late date, or risk further confusion, the specification is hereby changed as follows:

Change:

// Where appropriate, input is read directly into member variables.

To:

// Input is read directly into member variables, except for the title,

// which is read into a local String variable, processed if necessary,

// and then assigned to the title member variable.


Note: The specified behavior of getline for String is not changed.
