Component Demonstrations

  • Ordered_array demo1 - storing a set of pointers to C-strings in order (text)
  • Ordered_array demo2 - storing a set of ints in order (text)

The above are demos of how Ordered_array is supposed to be used. It illustrates how the Ordered_array functions can be used to create a set in sorted order and then print it out, and add and remove items from the set. They also show the use of function pointers and function pointer casts. A simple program like this is what you should use as a "test harness" to test your Ordered_array functions systematically. However, these are demos - not complete tests.

Behavior Samples

The following sample behavior files illustrate the program behavior and contain all of the different output message text strings produced by the program (see the strings.txt file). Your program should match these samples exactly so that you can be sure it is producing all of the different kinds of messages correctly. However, these samples are not a complete test of your program. Your program will be tested for a large number of other possible inputs and input sequences. The console interaction files are a capture of the console display during the program run. The stdin and stdout files are for you to use with redirected I/O. When given the stdin file, your program's stdout output should match the stdout file exactly. The other output files are always produced by the program, and your program should produce files that match exactly.

Exception to exact match: These samples were obtained with a 32-bit system. The number of bytes showing in the program's memory allocation output may be larger if you are using a 64-bit system. To see if you are matching the samples, either determine what the correct output is supposed to be for your system, or use compiler options to force a 32-bit compile and link. The values produced by a properly written program will be correct automatically on the autograder system.

Type-ahead & Errors Sample Behavior

The following sample provides an extreme example of the user putting in excess whitespace and "typing ahead" and also shows the error messages produced by the program and how the rest of the input line is ignored after an error message. This sample uses the small_dct.txt and small_crd.txt sample files for input. Your program should reproduce the stdout and the output files exactly.

Your program should get the whitespace-skipping and type-ahead capability by simple and idiomatic use of the stream input facilities; you should get it "automatically" - no special coding involved.

It is difficult to produce an error on opening a file for output. This sample uses ".." as an illegal output file name. Send for help if you have trouble with producing an output file opening error on your platform.
